바둑, 체스, 오목 같은 게임에서 "최선의 수"를 찾는 가장 확실한 방법은 모든 가능한 경우를 끝까지 탐색하는 것입니다. 하지만 게임의 복잡도는 상상을 초월합니다.

02 기존 접근: Minimax

전통적인 게임 AI는 Minimax + Alpha-Beta Pruning 방식이었습니다. "나는 최선을, 상대는 최악을 선택한다"고 가정하고 트리를 탐색하되, 가망 없는 가지를 잘라내는(pruning) 방식입니다.

체스에서는 성공

1997년, IBM의 Deep Blue가 세계 챔피언 카스파로프를 이긴 것이 이 계열입니다. 체스는 말의 점수, 위치 등으로 평가 함수를 만들 수 있었습니다.

바둑에서는 실패

하지만 바둑은 두 가지 문제가 있었습니다:

1. 평가 함수를 만들 수 없다 — 돌의 영향력, 세력, 집 등이 너무 복잡해서 "이 보드가 얼마나 유리한가"를 점수로 매기는 것이 극도로 어려움
2. 가지가 너무 많다 — 한 턴에 200수 이상 가능. 가지치기를 해도 감당 불가

결과적으로 바둑 AI는 수십 년간 아마추어 수준에 머물렀습니다.

Minimax의 근본적 한계: 인간이 설계한 평가 함수(evaluation function)에 의존한다는 것. 이 함수를 만들 수 없으면 무용지물입니다.

03 몬테카를로 방법

"Monte Carlo"는 랜덤 샘플링으로 답을 추정하는 방법론 전체를 가리킵니다. 이름은 모나코의 몬테카를로 카지노에서 따왔습니다 — 주사위를 많이 굴려서 확률을 추정한다는 비유입니다.

예시: π 추정하기

정사각형 안에 점을 랜덤으로 찍어서, 원 안에 들어간 비율로 π를 추정할 수 있습니다.

π ≈ ?
π = 4 × (원 안의 점 / 전체 점)
전체: 0
원 안: 0
실제 π: 3.14159...

게임에 적용하면?

"이 수를 둔 뒤 랜덤으로 끝까지 플레이해 보고, 이겼는지 졌는지 세자." 이걸 수천 번 반복하면, 승률이 높은 수 = 좋은 수라고 추정할 수 있습니다. 평가 함수가 전혀 필요 없습니다 — 끝까지 해보면 승패를 알 수 있으니까요.

2016

MCTS + 딥러닝. 랜덤 시뮬레이션을 신경망 평가로 대체. 이세돌 9단 격파.

2017

인간 기보 없이 자기 대국(self-play)만으로 학습. 바둑, 체스, 장기를 모두 정복.

07 UCB — Upper Confidence Bound

핵심 철학: "Optimism in the Face of Uncertainty"

불확실할수록 낙관적으로 추정합니다. 각 선택지의 진짜 가치를 모르니까, 관측된 평균에 "불확실성 보너스"를 더합니다.

UCB1(i) = X̄ᵢ + C × √(ln N / nᵢ)
X̄ᵢ = 평균 보상 (exploitation)  |  C × √(ln N / nᵢ) = 불확실성 보너스 (exploration)

이름의 의미

통계학의 신뢰구간(confidence interval)에서 옵니다. 동전을 10번 던져서 7번 앞면이 나왔다면, 진짜 확률의 95% 신뢰구간은 35%~93%입니다. 100번 던져서 70번이면 60%~79%로 좁아집니다.

UCB는 이 신뢰구간의 상한(upper bound)을 기준으로 선택합니다. 많이 시도한 쪽은 구간이 좁아서 상한이 평균에 가깝고, 적게 시도한 쪽은 구간이 넓어서 상한이 높습니다.

이론적 토대: Hoeffding 부등식

√(ln N / nᵢ) 항은 Hoeffding's inequality에서 유도됩니다.

P(|X̄ - μ| ≥ t) ≤ 2·exp(-2n·t²)
n번 독립 시행의 관측 평균 X̄가 진짜 평균 μ에서 t 이상 벗어날 확률의 상한

이 부등식의 조건은 딱 두 가지뿐입니다:

1. 시행이 독립일 것
2. 값이 유한한 범위 [a, b] 안에 있을 것

분포의 모양은 상관없습니다. 어떤 분포든 성립합니다. 그래서 UCB가 게임 종류에 관계없이 적용 가능한 이유이기도 합니다. 단, 조건이 느슨한 만큼 보수적인(느슨한) 부등식입니다.

Regret bound: UCB1의 regret은 O(ln T)로 증가합니다 (T = 총 시행 횟수). 이는 이론적으로 달성 가능한 최선의 증가율입니다. 즉, UCB1은 최적에 가까운 전략입니다.

08 Q-Value 변환

노트북 코드의 get_ucb()에서 q_value 변환이 왜 필요한지, 단계별로 봅니다.

def get_ucb(self, child):
    q_value = 1 - ((child.value_sum / child.visit_count) + 1) / 2
    return q_value + self.args['C'] * math.sqrt(math.log(self.visit_count) / child.visit_count)
STEP 1

원본 평균

v / n
범위: [-1, 1]

child 시점의 평균 가치.
-1 = 전패, 0 = 무승부, +1 = 전승

→
STEP 2

[0,1] 정규화

(v/n + 1) / 2
범위: [0, 1]

UCB1은 [0,1] 보상을 가정.
-1→0, 0→0.5, +1→1

→
STEP 3

시점 뒤집기

1 - (위의 값)
범위: [0, 1]

부모(상대) 시점으로 전환.
child 전패(0)→부모에겐 1(좋음)

왜 시점을 뒤집나? — 부모가 자식 중 하나를 선택하는 상황입니다. 자식 노드는 상대의 턴이므로, 자식이 이기면 부모에겐 나쁜 것입니다. 따라서 자식의 가치를 뒤집어야 부모 시점의 올바른 판단이 됩니다.

09 실전 디테일

미방문 자식 처리

visit_count = 0이면 UCB 공식의 분모가 0이라 계산 불가능합니다. 코드는 이 문제를 구조적으로 회피합니다:

select()는 is_fully_expanded()일 때만 호출됩니다 — 즉 모든 자식이 최소 1번은 방문된 상태에서만 UCB 비교가 일어납니다. 아직 안 만든 수가 남아 있으면 expand()에서 랜덤 하나를 골라 새 자식을 만듭니다.

따라서 초반에는 자식 수만큼 탐색하면 모든 자식을 한 번씩 거치게 되고, 그 과정은 완전 랜덤입니다. 그 이후부터 UCB 기반 선택이 시작됩니다.

메모리: 탐색한 노드만 저장

Expansion에서 한 번에 하나씩 추가하므로, N번 탐색 → 최대 N개 노드입니다 (선형). Simulation(랜덤 플레이) 과정은 트리에 기록하지 않고 버립니다.

10만 번 탐색해도 10만 개 노드 — 각 노드에 보드 상태 + visit_count + value_sum만 저장하면 됩니다.

이전 턴 결과 재활용

내가 수A를 뒀고, 상대가 수B로 응수했다면, 기존 트리에서 루트→A→B 경로의 서브트리를 새 루트로 재사용할 수 있습니다. 초반에는 가지가 많아 재활용 가능한 서브트리가 작을 수 있지만, 버리는 것보다는 낫기 때문에 대부분 재활용합니다.

같은 턴 내 순차적 누적

1000번 탐색은 독립이 아니라 순차적으로 누적됩니다. 매 탐색마다 트리 전체의 통계가 업데이트되고, 다음 탐색의 Selection이 그 통계를 즉시 반영합니다.

한 번 졌다고 영원히 안 가는 게 아니라, exploration 보너스 덕분에 다시 방문하게 되고, 그때 다른 결과가 나오면 평가가 수정됩니다.

10 AlphaGo로의 발전

기본 MCTS의 가장 큰 약점은 Simulation 단계입니다. 완전히 랜덤으로 끝까지 플레이하는 것은 특히 바둑처럼 복잡한 게임에서 부정확합니다.

기본 MCTS

Selection: UCB로 선택
Expansion: 자식 추가
Simulation: 랜덤 플레이 (rollout)
Backpropagation: 결과 전파

시뮬레이션이 랜덤이라 부정확. 바둑에서 한 턴에 수백 수를 랜덤으로 두면 의미 있는 결과를 기대하기 어려움.

AlphaGo / AlphaZero

Selection: PUCT (policy network 안내)
Expansion: 자식 추가
Simulation: 신경망이 가치 평가 (value network)
Backpropagation: 결과 전파

끝까지 안 가도 됨. 신경망이 "이 상태는 몇 점짜리"를 바로 평가. 어떤 수를 먼저 탐색할지도 신경망이 안내.

핵심 통찰: MCTS의 프레임워크는 그대로 유지하되, 가장 약한 고리(랜덤 시뮬레이션)를 신경망으로 교체한 것입니다. 이 프로젝트의 노트북 4.AlphaMCTS.ipynb이 바로 이 발전을 다룹니다.
PUCT(i) = Q(i) + C × P(i) × √N / (1 + nᵢ)
P(i) = policy network이 예측한 "이 수를 둘 확률". 신경망이 유망하다고 판단한 수에 exploration 보너스를 더 부여.
